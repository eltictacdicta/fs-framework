# Plan 01: Plugin Composer Setup

Give `api_base` an isolated Composer manifest and vendor tree for `zircote/swagger-php`, load plugin autoload from `config/services.php`, and document the DDEV install path. Do **not** remove packages from root `composer.json` (Phase 13).

## must_haves

- `plugins/api_base/composer.json` declares `zircote/swagger-php` ^6.0 and PSR-4 `FSFramework\Plugins\api_base\` → `Api/`
- `plugins/api_base/composer.lock` committed; `plugins/api_base/.gitignore` ignores `/vendor/` only (not lockfile)
- `plugins/api_base/vendor/autoload.php` exists after `ddev exec composer install --working-dir=plugins/api_base`
- `config/services.php` loads plugin vendor autoload **before** `api_base_load_legacy_models()` via `is_file` guard + `error_log` on missing vendor (D-04)
- `plugins/api_base/AGENTS.md` documents install command and missing-vendor behavior
- `class_exists('\\OpenApi\\Generator')` returns true after plugin services bootstrap (with vendor installed)
- Root `composer.json` unchanged in this phase (swagger-php remains until Phase 13)
- `ddev exec php vendor/bin/phpunit -c plugins/api_base/phpunit.xml` exits 0

## Verification

```bash
ddev exec composer install --working-dir=plugins/api_base
ddev exec php -r "require 'plugins/api_base/vendor/autoload.php'; exit(class_exists('OpenApi\\Generator') ? 0 : 1);"
ddev exec php vendor/bin/phpunit -c plugins/api_base/phpunit.xml
```

## <threat_model>
- **Threats:** Low — adding Composer manifest and loading vendor autoload. Supply-chain risk limited to Packagist `zircote/swagger-php` (same package already in root lockfile).
- **Data flow:** No change to auth tokens or user input handling.
- **ASVS L1:** V8.1.1 — use pinned lockfile (D-03); no new external URLs or secrets.
</threat_model>
