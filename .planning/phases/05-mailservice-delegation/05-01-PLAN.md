# Plan 01: empresa Mail Delegation to MailService

Replace direct PHPMailer usage in `empresa` model with delegation to `MailService` (already exists at `src/Core/MailService.php`, 490 lines, fully functional). The `empresa` model keeps its public API (`new_mail()`, `mail_connect()`, `can_send_mail()`) for backward compatibility but internally delegates to `MailService`.

## must_haves

- `plugins/business_data/model/empresa.php` — no `new PHPMailer()` calls remain; `new_mail()`, `mail_connect()`, `can_send_mail()` delegate to `MailService`
- `plugins/business_data/controller/admin_empresa.php` — `mail_test()` method works identically after delegation
- Public API preserved: `empresa->new_mail()`, `empresa->mail_connect($mail)`, `empresa->can_send_mail()` remain callable with same signatures
- `email_config` property harmonized: empresa reads from `MailService::getConfig()` instead of maintaining duplicate defaults

## Verification

```bash
ddev exec php vendor/bin/phpunit
```
Must pass with 0 new failures. Full suite result must match pre-change baseline (381 tests, 0 failures, 0 errors, 14 skipped).

## <threat_model>
- **Threats:** None — this is a delegation refactor. No new data handling, no new I/O.
- **Data flow:** Email config flows from `fs_var` → `MailService` → `empresa` (read-only). Password handling uses `fs_var::simple_get_decrypted()` which already exists in MailService. No plaintext password exposures added.
- **ASVS L1:** V2.1.1 (password security) — MailService already uses encrypted storage for SMTP passwords. This refactor removes a duplicate code path that could bypass the encryption.
</threat_model>
