# Plan 02-01: Add declare(strict_types=1) to Base Files

## Objective

Add `declare(strict_types=1)` incrementally to 15 simple base files — starting with the smallest files that have the fewest methods and least coupling to other classes. Strict types prevent type coercion bugs and make the codebase safer for future refactoring.

## must_haves

- truths:
  - `declare(strict_types=1)` must be the first statement after `<?php` (before comments)
  - Files are chosen by ascending complexity: fewest methods, fewest lines, least coupling
  - The test suite provides the safety net — run after each batch
  - AGENTS.md recommends `declare(strict_types=1)` for modern PHP files
- risks:
  - Files with method signatures lacking type hints may break if callers pass wrong types
  - Files that share global state with non-strict files may have implicit type coercion
  - The config.php file (0 lines, no classes) is excluded — it's a define() file
  - fs_autoload.php and fs_model.php are base classes depended on by everything — add strict_types LAST
  - The empty file (155 lines, 0 methods) on row is actually not PHP — skip

## Tasks

### Task 1: Batch 1 — Smallest files (6 files, <100 lines each)

<read_first>
- base/fs_secret_migrator.php
- base/fs_log_manager.php
- base/fs_list_filter.php
- base/fs_list_filter_checkbox.php
- base/fs_list_filter_select.php
- base/fs_list_filter_date.php
</read_first>

<action>
For each file listed above, add `declare(strict_types=1);` on a new line immediately after `<?php`:
```php
<?php
declare(strict_types=1);
```
Keep the existing file header comment and all other code unchanged. The `declare(strict_types=1)` statement must appear before any non-docblock comment, use, or require statement.
</action>

<acceptance_criteria>
- Each of the 6 files contains `declare(strict_types=1);` on line 2 or 3 (after `<?php`)
- Each file passes `php -l` (no syntax errors)
- `ddev exec php vendor/bin/phpunit --testsuite Base` passes
</acceptance_criteria>
