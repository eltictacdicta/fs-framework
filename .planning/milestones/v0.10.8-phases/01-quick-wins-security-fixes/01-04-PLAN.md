# Plan 01-04: Replace @ Error Suppression in base/

## Objective

Replace all `@` error suppression operators in `base/` files with proper error handling: explicit checks before operations, try/catch blocks, or error_get_last() patterns. This ensures real errors (disk full, permission denied, corrupt cache) are visible for debugging.

## must_haves

- truths:
  - `@` suppresses ALL errors including fatal ones — this hides real problems
  - Most `@` usage is on filesystem operations (mkdir, unlink, file_put_contents, rename)
  - The pattern `@mkdir($path, 0755, true)` should become `if (!is_dir($path) && !mkdir($path, 0755, true) && !is_dir($path)) { error_log(...); }`
  - The pattern `@unlink($path)` should become `if (file_exists($path) && !unlink($path)) { error_log(...); }` or just `file_exists($path) && unlink($path)` where failure is acceptable
  - The pattern `@file_put_contents(...)` should become explicit error checking with error_log on failure
  - `@rename()` should check return value and log failure
  - `@copy()` should check return value and log failure
- risks:
  - Changing error handling behavior could surface previously-hidden warnings in production
  - Some `@` usage may be intentional (e.g., unlink on files that may not exist) — use `file_exists()` guard instead of try/catch
  - Must not break the file-based cache, plugin manager, or upload functionality
