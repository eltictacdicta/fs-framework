# Milestone v0.10.8: Tech Debt Cleanup

**Status:** ✅ SHIPPED 2026-05-16
**Phases:** 1-3
**Total Plans:** 9

## Overview

Incremental cleanup of legacy technical debt in FSFramework. Three phases moved from low-risk quick fixes (version guards, dead code removal) through type safety hardening to structural decompositions. Each phase was independently revertible with full test suite verification.

## Phases

### Phase 1: Quick Wins & Security Fixes

**Goal**: Entry points are correct, dead security code is removed, only one PHPMailer version exists, and error suppression is eliminated
**Depends on**: Nothing (first phase)
**Plans**: 4 plans

Plans:

- [x] 01-01: Update PHP version guards in index.php and install.php to 8.2 [REQ-01]
- [x] 01-02: Remove weak random fallback (str_shuffle) from install.php [REQ-03]
- [x] 01-03: Remove PHPMailer 5.x vendored code and compat bridge [REQ-04]
- [x] 01-04: Replace @ error suppression with proper error handling in base/ [REQ-08]

**Details:**
- `index.php` and `install.php` check for PHP >= 8.2 using `version_compare()` instead of float casting
- `install.php` uses only `random_bytes()` — no `str_shuffle()` fallback
- `extras/phpmailer/` (61 files) and `phpmailer_compat.php` deleted; all email uses Composer PHPMailer 6.x
- All `@` error suppression replaced with proper guards in 11 base/ files (~35 ops)
- Base test suite: 124/124 pass, zero regressions

### Phase 2: Type Safety & Test Coverage

**Goal**: Core files have strict type enforcement and business-critical plugins have automated tests
**Depends on**: Phase 1
**Plans**: 2 plans

Plans:

- [x] 02-01: Add declare(strict_types=1) to 15 base files (3 batches by complexity)
- [x] 02-02: Add test coverage for business_data (empresa, ejercicio, serie, divisa) and catalogo_core (familia, fabricante)

**Details:**
- 15 base files with `declare(strict_types=1)` (exceeding 10-file minimum)
- Fixed `trim(fgets())` bug in `fs_ip_filter.php` exposed by strict_types
- 14 new plugin tests: catalogo_core (familia, fabricante) + business_data (empresa, ejercicio, serie, divisa)
- Plugin independence confirmed: catalogo_core autonomous, business_data independent
- Base tests: 124/124 pass

### Phase 3: Structural Decomposition

**Goal**: SHA1 legacy code lives only in legacy_support plugin, and monolithic classes are split into focused services
**Depends on**: Phase 2
**Plans**: 3 plans

Plans:

- [x] 03-01: Delegate SHA1 password verification entirely to legacy_support plugin
- [x] 03-02: Decompose StealthMode into CssSanitizer + HtmlSanitizer + thin facade
- [x] 03-03: Extract FsMysqlSchemaUtility with normalization methods

**Details:**
- SHA1/MD5 verification removed from `PasswordHasherService`; `legacy_support` now sole owner
- `isLowercasedLegacySha1Bypass()` kept in core as security gate
- `StealthMode` reduced from 1073 → 508 lines (CssSanitizer 246 + HtmlSanitizer 388 extracted)
- `FsMysqlSchemaUtility` with 3 normalization methods extracted from `fs_mysql`
- admin_cache controller deferred — cache code in admin_home too trivial (5 lines)
- Base tests: 124/124 pass throughout

## Milestone Summary

**Key Decisions:**
- Use `version_compare()` instead of float casting for PHP version checks (Phase 1)
- Proxy pattern confirmed: business_data delegates domain models to catalogo_core (Phase 1)
- SHA1/MD5 verification lives only in legacy_support plugin (Phase 3)
- StealthMode decomposition: facade pattern preserves backward compatibility (Phase 3)

**Issues Resolved:**
- Incorrect PHP 5.6 version guards in entry points
- Dual PHPMailer (5.x vendored + 6.x Composer) causing maintenance burden
- Widespread `@` error suppression hiding real disk/permission errors
- 5 untested plugins with zero test coverage
- 43 base files missing `declare(strict_types=1)`
- SHA1/MD5 password verification logic duplicated between core and legacy_support
- 1073-line monolithic StealthMode class

**Issues Deferred:**
- `empresa.php` should delegate email configuration to `MailService` instead of instantiating PHPMailer directly
- Deeper `fs_mysql` decomposition (20+ interdependent schema methods)
- Plugin management extraction from `admin_home`
- Database migration system (separate initiative)
- Queue/job system (infrastructure change)
- Full Twig migration (ongoing)

**Technical Debt Incurred:**
- `HtmlSanitizer` at 388 lines slightly exceeds 300-line target due to rendering helpers included
- `StealthMode` at 508 lines (vs. <200 aspirational target) — access/login logic is tightly coupled
- 5 pre-existing test failures in Security/SessionManager and Cache/DataSrcRepository remain
